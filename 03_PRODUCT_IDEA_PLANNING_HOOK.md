# Product / Idea Planning Hook

## Scope

Use for product ideas, startup ideas, feature planning, MVP design, idea validation, roadmap slicing, user problem definition, and service concept refinement.

## Purpose

Prevent attractive but unfocused product ideas with no user, no wedge, no validation, no distribution, and no execution path.

## Common Weak Patterns

- “AI-powered platform” with no input -> transformation -> output -> user action.
- Feature list instead of user pain.
- Roadmap before MVP.
- No non-goals.
- Generic persona such as “busy professionals.”
- No current alternative or switching trigger.
- No first 10 users plan.
- No pricing/distribution assumption.
- Idea requires scale before value appears.

## Safety Boundary

- Do not propose deceptive, privacy-invasive, illegal, or exploitative products.
- Do not fabricate market validation.
- When uncertain, design tests instead of inventing proof.

## Required Output Contract

Include:

1. **One-Line Product Thesis**
   - narrow user,
   - painful problem,
   - promised outcome,
   - differentiated approach.

2. **Target Segment**
   - narrow ICP,
   - excluded users,
   - why this segment first.

3. **Problem and Current Alternative**
   - current behavior,
   - pain intensity,
   - switching trigger,
   - why current solutions fail.

4. **Wedge**
   - first sharp use case,
   - why it can spread,
   - why competitors may miss it,
   - unfair advantage if any.

5. **MVP Scope**
   - 3-5 must-have items,
   - explicit non-goals,
   - manual/AI-assisted shortcuts,
   - v0 data/input/output contract.

6. **First 10 Users Plan**
   - where to find them,
   - outreach script or message,
   - artifact to show,
   - response that counts as interest.

7. **Validation Plan**
   - fastest test,
   - success metric,
   - failure/kill criterion,
   - timeline.

8. **Roadmap**
   - v0,
   - v1,
   - later only if validated.

## AI Platform Translation Rule

If “AI platform,” “AI automation,” or “AI assistant” appears, translate it into:

```text
input -> AI transformation -> output -> user decision/action -> measurable value
```

## Artifact Boundary

When organizing a product docs repository, create or keep only documents that a founder, PM, engineer, operator, designer, or reviewer would actually read to understand the product and execute work.

Do not add separate files that exist mainly to explain PM documentation theory, Codex's document-cleanup rules, source-reading methodology, 작업자용 규칙, answer-quality standards, hook compliance notes, or why a particular documentation framework was used. Put those rules in this hook, another hook, a skill, or summarize them in the assistant response.

Allowed product-doc artifacts include strategy, PRD, validation plan, roadmap, backlog, open questions, operating model, protocol, technical design, decision log, glossary, and runbook. Disallowed project-doc artifacts include meta standards for Codex, generic “how to write docs” notes, and external-research summaries that do not change product decisions.

## Forbidden Output Patterns

Invalid final answers:

- broad platform idea with no first use case,
- ten features with no sequence,
- “build community” with no acquisition mechanic,
- “collect feedback” with no script or metric,
- “use AI to automate” with no input/output contract,
- idea requiring scale before value.

## Strong Answer Shape

```text
제품 한 줄: [ICP + problem + outcome]
첫 타깃/제외 타깃: [narrow user / non-users]
현재 대체재: [what they do now]
첫 wedge: [specific use case]
MVP 포함/제외: [must-have / non-goals]
첫 10명 확보: [where + message + artifact]
검증 실험: [test]
성공/실패 기준: [metric / kill]
다음 7일: [steps]
```

## Final Gate

Before finalizing, confirm:

- the ICP is narrow,
- the first wedge is clear,
- non-goals are explicit,
- first 10 users plan exists,
- validation has metric and kill criterion,
- AI/platform language is translated into input/output/action.

If missing, rewrite before finalizing.
